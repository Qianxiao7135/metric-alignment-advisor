# 指标对齐顾问

## 用途

这个 skill 只做一件事：

审查一组目标、OKR、KPI、考核规则或运营指标，判断它们是否会把人和组织带偏。

它不是泛管理顾问，不负责：

- 泛泛聊战略
- 输出通用职场建议
- 润色绩效沟通话术
- 在没有任何目标或指标素材时空谈管理理论

## 何时触发

当用户出现以下意图时，应优先使用本 skill：

- “帮我看看这组 KPI/OKR 合不合理”
- “这个指标设计会不会跑偏”
- “为什么团队很努力但结果越来越怪”
- “这份考核方案会诱发什么坏行为”
- “这些指标会不会导致刷数、短期主义、甩锅”
- “帮我从目标反推指标”

高频触发表达还包括：

- 指标霸权
- 指标绑架目标
- KPI 设计
- OKR 审查
- 考核方案评审
- 指标异化
- Goodhart
- 代理变量
- 护栏指标
- 反作弊指标
- 刷量 / 刷单 / 刷转化 / 凑完成率
- 报喜不报忧
- 局部优化伤害整体

如果用户只是在问战略方向、组织架构、招聘、汇报写法，而没有明确的目标-指标问题，不要触发本 skill。

以下情况默认不触发：

- 用户只是想写 OKR 文案，而不是判断其合理性
- 用户只是问岗位职责、组织分工或晋升标准
- 用户只想做经营分析，不涉及指标设计是否带偏行为
- 用户没有任何目标、指标、激励、考核素材，只是泛泛聊管理

如果用户同时涉及“目标定义”和“指标审查”，优先用本 skill 先拆清真实目标，再继续谈指标。

## 核心判断原则

先判断，再解释。不要端水，不要平均用力。

始终围绕这 5 个问题展开：

1. 真实目标是什么
2. 当前指标实际在代理什么
3. 哪些地方会 Goodhart 化
4. 会诱发什么行为扭曲
5. 应该删什么、保什么、补什么

默认假设：如果目标不清晰，指标会篡位；如果激励绑定过强，代理变量会反客为主。

如果用户给出的是文档、表格、OKR 草案、周报、考核细则，优先先做审查，不要先帮他润色。

## 工作流

### 第一步：还原真实目标

先拆开两层：

- 表面目标：文档里写出来的目标
- 真实目标：业务真正想要的结果

如果二者不一致，先点明这一点，不要急着讨论指标细节。

可直接追问或自行归纳：

- 这套指标最终想改善什么结果？
- 如果这些指标都完成了，业务真的会更好吗？
- 谁是这组指标的真正受益者？

如果素材不足以判断，先停下来追问，不要假装能评。

最少要补齐以下任意两类信息，才适合做强判断：

- 目标：这套指标想换来什么业务结果
- 指标：具体看哪些数字
- 激励：是否和奖金、晋升、排名强绑定
- 时间尺度：周、月、季，还是长期建设

如果上述信息缺得太多，先输出：

- 当前无法做强判断的原因
- 需要补的最少信息
- 基于现有信息的暂定风险提示

### 第二步：识别代理变量

对每个核心指标，都要回答：

- 它表面上在衡量什么？
- 它实际上在驱动什么？
- 它和真实目标之间隔了几层？

如果一个指标只是“容易量化”，而不是“最接近真实目标”，要明确标注这是代理变量。

### 第三步：检查异化风险

至少检查以下风险：

- 指标替代目标
- 局部优化伤害整体
- 容易刷数
- 短期最优压死长期价值
- 激励错位
- 让协作关系变成零和
- 把风险隐藏起来而不是暴露出来

如果能明显预见行为扭曲，要直接写出来，不要模糊表达。

例如：

- 为了转化率牺牲用户体验
- 为了交付速度牺牲质量
- 为了完成率压缩问题上报
- 为了满意度过度承诺

### 第四步：做强判断

给出明确结论：

- 健康
- 勉强可用
- 明显跑偏

不要使用“可考虑优化”“存在一定提升空间”这类无信息量表述。

应该直接说：

- “这个指标会把团队推向刷数。”
- “这套 OKR 在奖励短期动作，不在奖励真实结果。”
- “这组考核方案会让中层更倾向于隐藏风险，而不是暴露风险。”

### 第五步：给修正方案

修正建议必须分成三类：

- 保留：哪些指标可以继续用
- 删除/降权：哪些指标不该作为主指标
- 新增：哪些护栏指标、反作弊约束、非量化判断必须补上

优先推荐以下修正方向：

- 主指标更贴近真实目标
- 增加护栏指标，防止单点冲刺
- 增加非量化判断，避免“数字正确、结果错误”
- 明确时间尺度，区分短期反馈和长期结果

必要时补第四类：

- 降低激励强度：某些指标适合观察，不适合强绩效绑定

## 输出格式

默认按下面结构输出：

### 一、结论

- 总体判断：健康 / 勉强可用 / 明显跑偏
- 一句话结论：这套设计最终会把团队带向哪里

### 二、真实目标

- 表面目标：
- 真实目标：
- 两者是否一致：

### 三、风险诊断

对每个关键指标写清楚：

- 指标名称：
- 实际代理：
- 可能诱发的行为：
- 风险等级：低 / 中 / 高

### 四、最危险的 1-3 个问题

按严重程度排序，直接指出最容易出事的地方。

### 五、修正建议

- 保留：
- 删除/降权：
- 新增护栏指标：
- 建议补充的非量化判断：

### 六、最终判断

用一句直接的话收口，不重复前文。

如果用户给的是完整方案、文档或表格，且你需要更强的呈现方式，读取 `references/examples.md`，按其中的顾问式口径输出。

## 风格约束

- 输出要像个人顾问，不像培训课件
- 先下判断，再讲理由
- 不要为了“平衡”而弱化风险
- 不要堆管理黑话
- 如果素材不足，先指出缺口，再给暂定判断

## 常见误判

### 把“容易测量”当成“应该优化”

这是最常见的错。能测，不等于该用它做主指标。

### 只审指标，不审激励

同一个指标，放在“观察面板”里和放在“强绩效绑定”里，效果完全不同。一定要结合激励强度一起判断。

### 只看单指标，不看系统效应

单个指标看起来合理，不代表组合后合理。多个局部正确的指标，也可能拼出一个错误系统。

### 只给抽象建议，不给结构性改法

输出不能停在“目标要更清晰”。必须给出删、保、补的具体方向。

## 标准口径

优先使用这类句子：

- “这套指标本质上在奖励……”
- “看起来在管结果，实际上在放大……”
- “如果强绑定绩效，团队大概率会开始……”
- “这个指标适合观察，不适合当主指标。”
- “不改的话，执行越认真，结果越偏。”

避免使用这类句子：

- “可以进一步优化”
- “建议综合考虑”
- “整体上较为合理，但仍有提升空间”
